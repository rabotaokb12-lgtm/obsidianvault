# 1. Linux #linux
Надо знать везде потому как большинство сервисов и серверов работают на нём:
- Настройки сервис, прав доступа, просмотр логов, автоматизация в среде Linux 
- Управление процессами, хранение логов, настройка сетевых параметров и т.д
## Какие аспекты Linux важны: 
### 1.1 Повседневная работа с терминалом 
- Базовый команды: ls, cd, mkdir, rm, cp, mv, sudo 
- Фильтрация и поиск: grep, find. xargs, sort, awk, sed 
	**grep** - поиск по тексту : grep "error" /var/log/syslog #grep 
	**find** - поиск файлов и директорий(по имени по типу, размеру) : find /home -name "* .txt"  #find
	**xargs** - передаёт аргументы в команду, берёт входящие данные (обычно из grep find) и подставляет аргументы : find /tmp -name "* .log" | xargs rm #xargs
	**sort** - сортировка строк : sort users.txt(сортировка по алфавиту) #sort
	**awk** - обработчик текста по колонкам, работает со столбцами, поддерживает условия, арифметику : awk '{print $2}' file.txt (вывести вторую колонку) #awk
	**sed** - потоковый редактор(замена и удаление текста) : sed 's/Hello/Hi/g' file.txt (Заменить Hello на Hi) #sed 
- Понимание структуры каталогов: /etc, /var/log, /usr, /home и т.д

### 1.2 Пользователи права и группы  chmod stat ls -l chown chgrp usermod
- Как создать пользователя(useradd), как задать пароль, как добавить в группу(groupadd, usermod)
	``sudo useradd username -m -s /bin/bash username``:
		-m - создать домашнюю директорию /home/username 
		-s /bin/bash - указать что чел будет использовать стандартный bash(чтобы мог юзать терминал) 
	``sudo passwd username`` - создание пароля
	``sudo groupadd developers`` - создание новой группы
	``sudo usermod -aG developers username`` - добавить пользователя в новую группу
		-ag - добавить доп группы не перезаписывая старые 
- Что такое chmod, chown, umask и как управлять правами в Linux 
	**chmod** 755 file - изменение прав доступа 
		4 - чтение - r 
		2 - запись - w 
		1 - выполнение - x 
		Вот от их сложений и появляются разные права например 777 или 755 или 765 А ка
		`chmod u+x` - добавить владельцу право на выполнение
		`g` - группы 
		`o` - остальные
		`a` - всем
	**chown** user file - сменить владельца 
		chown user:group file - владелец и группа 
		chown :group file - только группа 
		chown -R user:group dir/ - рекурсивно(Всё что находится в директорий)
	**chgrp** - изменить группу файла
		`chgrp name_of_group filename` 
	**umask** 022 - убирает права для всех новых файлов 
		В данном случае все новые файлы будут получать права доступа 644(т.к по умолчанию 666)
		Директорий новые останутся в правах 755 в ЛЮБОМ случае
- `stat имя файла` - проверки прав доступа, владельца и группы 1 файла
  `stat путь до директорий/*` - чекнуть права владельца и группу директорий 
	  Access: (0775/drwxrwxr-x)  Uid: ( 1000/   usern)   Gid: ( 1000/   usern)
	  uid-владелец
	  gid -группа
- `ls -l имя файла` - посмотреть права определённого файла 
- `ls -ld имя директорий` - чекнуть права директорий
### 1.3 Управление сервисами (Systemd, SysV init)
- Systemd: сервисные юниты(.service), логи (journalctl), команды (systemctl start/stop/restart)
	**Systemd** - система инициализаций и управления сервисами в Линух(раньше был init)  #systemd
		Запуск и остановка сервисов при старте/остановке системы
		управление зависимостями между сервисами 
		Логирования(через journal) 
		Отслеживание процессов и автоматический рестарт при падений 
		То есть **systemd** - это менеджер всего, что работает в системе.
	**Юниты** - это файлы конфигураций, которые описывают, как **systemd** управляет сервисами и другими объектами. 
		Основные типы юнитов: 
			.service - Сервисы (nginx, sshd, docker)
			.socket - Сокеты, через которые запускаются сервисы
			.target - Группы юнитов (аналог runlevel)
			.timer - Планировщики, вместо cron
			.mount - Точки монтирования
			.automount - Автоматическое монтирование
			.device - Аппаратные устройства
		Сервисные юниты **.service**
			Обычно лежат в /etc/systemd/system/ 
	**journalctl** - это инструмент просмотра логов systemd. храниться в binary journal #journalctl
		Команды: 
			``journalctl`` - Все логи системы
			``journalctl -u nginx.service`` - логи конкретного сервиса 
			``journalctl -f`` - хвост логов в реальном времени
			``journalctl --since "2025-12-01"`` - логи с конкретной даты
			``journalctl -xe`` - расширенный вывод ошибок
### 1.4 Процессы и ресурсы
- Какие процессы занимают CPU. RAM(ps, top, htop)
	**ps** - показывает список процессов #ps 
		``ps aux`` - все процессы
		``ps aux | grep nginx`` - поиск процесса 
		``ps -u username`` - процессы пользователя
		``ps -p PID`` - информация о конкретном процессе 
	**top** - показывает процессы в реальном времени 
		Основные клавиши:
		`P` — сортировать по CPU
		`M` — сортировать по памяти
		`k` — убить процесс (ввести PID)
		`q` — выйти
	**htop** - улучшенная версия top 
		`F6` — сортировка
		`F4` — фильтр
		`F9` — kill процесс
		`F5` — tree (древо процессов)

### 1.5 Логи и журналы #logs 
- Просмотр:
	`cat /var/log/syslog`  `/var/log/messages` `/var/log/nginx/error.log`:
		Через head tail less
			**head** - первый 10 строк, для большего кол-во ``head -n 50`` #head
			**tail** - последние 10 строк. для большего кол-во ``tail -n 50`` #tail
			**less** - программа для удобного просмотра больших файлов #less
				**стрелки / PgUp / PgDn** — листать
				**q** — выйти
				**/** текст → поиск по файлу
				**n** — следующее совпадение
				**g** — в начало
				**G** — в конец
	`/var/log/journal/...` через **journalctl** #journalctl 


# 2. Сети #network
Надо знать потому как Devops настривает: 
- Доступ к сервисам(какие порты открыть? как прописать домен?)
- Контейнеры и виртуалки(каждому нужен IP адрес и маршрут)
- DNS-записи(поднять myapp.py, сделать "прокладку" через Nginx и SSL)
### 2.1 IP-адреса, порты и маршруты
- Порты (TCP UDP)
	HTTP(80) HTTPS(83) SSH(22) - классика
	Если порт не открыт в фаирволе доступ извне не будет
	tracepath домен - показывает путь до айпи

### 2.2 DNS 
Утилиты:
	nslookup днс имя - узнать ip сайта
	dig днс имя - более подробная инфа об этом днс имени

### 2.3 Фаирволы и ограничения доступа
Команды в линух: 
	iptables:
	nftables
	firewalld
	ufw 
	Забей можно загуглить сложные блять команды

Просто полезные команды
- `ping`
    - Проверяет, можно ли “достучаться” до IP, есть ли задержки, потери пакетов.
- `traceroute` / `tracepath`
    - Показывают маршрут, через какие узлы идёт трафик.
    - _Пример_: “Если на третьем узле теряются пакеты, там может быть проблема.”
- `telnet` или `nc (netcat)`
    - Проверить доступ к порту: “telnet 1.2.3.4 80” — если получилось, значит порт открыт, сервис
# 5 CI/CD #ci/cd
### Continuous Integration (CI)?
CI - непрерывная интеграция
	Когда любой новый код разработчика (commit. merge) автоматом проверяется 
	Система(Jenkins. Gitlab CI. Github Actions) скачивает свежий код, собирает, запускает тесты
### Continuous Delivery / Deployment (CD)?
C Delivery
	Код после тестов готов к релизу - но финальное решение нажимает человек (DevOps)
C Deployment
	Либо же код после всех проверок автоматический уходит в прод без подтверждения 
### Что делает Девопс 
1. Пишет пайплайн 
	В Jenkins(Jenkinsfile), GitLab(.gitlab-ci.yml), GitHub Actions(.yaml)
2. Автоматизирует тестирование
	Юнит тесты(проверка отдельных функций или модулей), Интеграционные тесты(разные части системы работают вместе),
	Линтеры(проверка кода на ошибки, стили и плохие практики
	Security-сканы(ищет уязвимости и небезопасные зависимости)
3. Настройка окружения 
	Часто бывает dev->test->stage->prod. 
	На каждом этапе свои проверки, переход между стадиями
4. Деплой
	Запуск Docker-контейнеров, выкладка в Kubernetes, копирование артефактов на сервера, обновление конфигов
### Пример процесса CI/CD 
1. Commit - разраб пушит код в репозиторий (Git)
2. Сборка(build) - CI-сервер подхватывает код, устанавливает зависимости, компилирует
3. Тесты - прогон юнит тестов, интеграционных тестов, возможно статистических анализаторов (SonarQube, ESLint)
	Статистические анализаторы - это проги которые проверяют код не запуская его. 
	ESLint - Для JavaScript/TypeScript. Типо **линтера** 
	SonarQube - проверяет на дублирование, уязвимости, тех долг, вонючий код. Поддерживает много языков
4. Сборка Docker образа(если надо) и загрузка его в registry 
	registry - хранилище докер образов
5. Деплой на тестовое окружение или сразу в прод 
6. Мониторинг
