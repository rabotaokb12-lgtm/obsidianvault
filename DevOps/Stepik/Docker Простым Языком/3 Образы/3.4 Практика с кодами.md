## 1 Python Flask 
**app.py**: 
```
from flask import Flask

app = Flask(__name__)

@app.route("/")
def hello():
    return "Hello from Docker!"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=8000)
```

**DockerFile**: 
```
FROM python:3.9
RUN pip install --no-cache-dir flask werkzeug==2.2.2.
COPY app.py /app/
WORKDIR /app
EXPOSE 8000
CMD ["python", "app.py"]`
```
flask - python веб сервер 
werkzeurg - библиотека на которой построен Flask 
--no-cache-dir - убирает кэш пакета 
**Команды**
`docker build -t apppython .`
`docker run -d -p 8000:8000 --name appbek apppython:latest`

```json
{
  "name": "my-node-docker-app",

  "dependencies": {
    "express": "^4.18.2"
  }

```

**index.js**
```json
{
  "name": "my-node-docker-app",
  "version": "1.0.0",
  "description": "Пример Docker-приложения на Node.js",
  "main": "index.js",
  "scripts": {
    "start": "node index.js"
  },
  "author": "",
  "license": "ISC",

  }

```

**dockerfile**
``` json
FROM node:16
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm install
COPY index.js ./
ENV PORT=3000
EXPOSE 3000
CMD ["npm", "start"]
```
Объясняю про последовательность строка 3 4 5 
1. Копируем сперва json-ы ведь в нём написана строки:
	"dependencies": {
	    "express": "^4.18.2"
2. Только потом устанавливаем зависимость 
   npm install смотрит на эти строки в json-e и понимает какую именно зависимость ему устанавливать 
3. Потом копируем index.js ведь если поставить его выше RUN то при КАЖДОМ изменений index.js зависимость будет снова устанавливаться

## 3 Alpine 
**entrypoint.sh** 
```bash
#!/bin/sh
echo "Starting ping script..."

# Если переменная не задана, берём дефолт
URL="${PING_URL:-https://example.com}"

while true
do
  echo "Pinging $URL..."
  curl -I $URL
  sleep 5
done
```

**dockerfile**:
``` bash
FROM alpine:3.17
RUN apk add --no-cache curl
COPY entrypoint.sh /app/entrypoint.sh
ENV PING_URL=https://exampe.com
ENTRYPOINT ["sh", "/app/entrypoint.sh"]
```
ЗАЧЕМ НУЖЕН sh в ENTRYPOINT чтобы не указывать право на выполнение 
А так если без него то нужно явно вписать права:
``` node
RUN chmod +x /app/entrypoint.sh
ENTRYPOINT ["/app/entrypoint.sh"]
```
## 4.Jar файл SpringBoot
### DemoApplication.java:
``` typescript 
package com.example;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@SpringBootApplication
public class DemoApplication {

    @GetMapping("/")
    public String hello() {
        return "Hello from Spring Boot!";
    }

    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}
```
#### Разбор кода

`package com.example;` - Определяет пакет (пространство имён) класса. Папка проекта должна совпадать:
`src/main/java/com/example/`

`import ...`
Подключение классов из Spring Boot:
- `SpringApplication` — запускает приложение
- `SpringBootApplication` — включает автоконфигурацию Spring
- `RestController` — делает класс веб-контроллером
- `GetMapping` — создаёт HTTP маршрут

`@RestController`
Говорит Spring:
Этот класс обрабатывает HTTP-запросы.
Методы начинают отвечать на веб-запросы.

`@SpringBootApplication`
Главная аннотация Spring Boot, включает:
- автоконфигурацию
- сканирование компонентов
- запуск приложения

`@GetMapping("/")`
Создаёт маршрут:`GET /`
Т.е. при открытии сайта вызывается этот метод.

Метод `hello()`
`public String hello()`-Возвращает текст, который отправляется в браузер.

`main()`
`SpringApplication.run(...)` - Точка входа — запускает приложение и веб-сервер.
### buid.gradle
Файл управляет:
- сборкой проекта,
- зависимостями,
- версией Java,
- созданием JAR.
Gradle по этому файлу понимает, **как собрать приложение**.
``` typescript
plugins {
    id 'java'
    id 'org.springframework.boot' version '3.2.5'
    id 'io.spring.dependency-management' version '1.1.4'
}

group = 'com.example'
version = '1.0'
java.sourceCompatibility = JavaVersion.VERSION_17

repositories {
    mavenCentral()
}

dependencies {
    implementation 'org.springframework.boot:spring-boot-starter-web'
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

tasks.named('test') {
    useJUnitPlatform()
}
```
##### plugins
Подключает плагины сборки.
`plugins {     id 'java'     id 'org.springframework.boot' version '3.2.5'     id 'io.spring.dependency-management' version '1.1.4' }`

`id 'java'`
Добавляет поддержку Java:
- компиляция `.java`
- создание `.jar`
`org.springframework.boot`
Плагин Spring Boot:
- собирает запускаемый JAR
- добавляет сервер
- настраивает запуск приложения
dependency-management
Управляет версиями библиотек автоматически, чтобы не было конфликтов.
##### group
`group = 'com.example'`
Идентификатор проекта (обычно имя компании или проекта).
##### version
`version = '1.0'`
Версия приложения.
Влияет на имя JAR:
`app-1.0.jar`
##### версия Java
`java.sourceCompatibility = JavaVersion.VERSION_17`
Говорит Gradle:
> Компилировать под Java 17.
##### repositories
Откуда скачивать библиотеки.
`repositories {     mavenCentral() }`
mavenCentral
Главное хранилище Java-библиотек в интернете.
##### dependencies
Библиотеки, которые нужны проекту.
`dependencies {     implementation 'org.springframework.boot:spring-boot-starter-web'     testImplementation 'org.springframework.boot:spring-boot-starter-test' }`
**starter-web**
Добавляет:
- веб-сервер
- REST API
- Spring MVC
- JSON поддержку
Без него HTTP сервер не появится.
**starter-test**
Библиотеки для тестирования приложения.
##### настройка тестов
`tasks.named('test') {     useJUnitPlatform() }`
Говорит Gradle запускать тесты через JUnit 5.
