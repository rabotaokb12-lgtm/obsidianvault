## 1. Ставить редко меняющиеся команды выше в Dokcerfile
Надо это т.к если поменять что то сверху то всё что ниже будет заново пересобираться. Докер хранит кэши и если сборка была проведена то если ничего не поменялось она не будет заново всё делать
Прим:
``` dockerfile
COPY . /app #ОЧЕНЬ ПЛОХО
RUN npm install

RUN npm install
COPY . /app #Ахуенно
```
## 2. Команда RUN - всё вместе
Каждая команда RUN создаёт отдельные слои, а это плохо
``` dockerfile
RUN apt-get update
RUN apt-get install -y curl 
RUN rm -rf /var/lib/art/lists/* #Хуёвая практика

RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/* #Пиздато так лучше
```
## 3. Минимализировать мусор 
В Линукс при скачиваний (apt-get install, apt-get update) возникают кеши, временные файлы. Их стоит очищать 
```swift
RUN apt-get update && apt-get install -y curl && \
    rm -rf /var/lib/apt/lists/*
```
## 4. Multi-stage builds
Это разделение сборки и запуска, А НАХУЯ? :
1. Уменьшает размер образа, т.к в образе не остаются 
	1. Компиляторы(программы что делают из кода исполняемый файл)
	2. **Dev зависимости** - библиотеки что нужно только для разработки(тестовые библиотеки(проверяльщики работоспособности кода), линтеры(проги для проверки кода на ошибки и стиль написания), инструменты сборки )
	3. **Dev tools** - проги что помогают собирать или разрабатывать приложения(git, curl, make, gcc)
2. Разделяет сборку и запуск
	1. **Build stage** - сборка проекта, установка dev tools, компиляция 
	2. **Production stage** - только запуск приложения
3. Повысить безопасность 
	1. Меньше иструментов в контейнере
	2. Меньше возможности для атак 
Если нужна комплиляция программы то **Multi stage** обязателен
Прим:
```sql
# Пример (Go), упрощённо
FROM golang:1.18-alpine AS build
WORKDIR /app
COPY . .
RUN go build -o myapp

FROM alpine:3.17
COPY --from=build /app/myapp /usr/local/bin/myapp
CMD ["myapp"]
```
