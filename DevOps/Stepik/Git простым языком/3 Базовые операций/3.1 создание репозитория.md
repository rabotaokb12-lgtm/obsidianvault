### 1 способ - инициализация
`git init` - создаёт в текущей дерикторий папку `.git`, которая содержит всю инфру для работы **Git**
	Инициализация создаёт пустой репозиторий, файлы проекта нужно будет явно добавить
### 2 способ - клоирование
`git clone`
- Создаётся новая директория с именем проекта
- Инициализируется Git-репа в нём
- Загружается вся история коммитов
- Создаётся рабочая копия последней версий файлов
Для клонирования Git используется след протоколы: HTTPS, SSH(требуется настройка ключей), Git-протокол и локальные пути
### Базовый конфиг
`git config --global user.name "Имя"`
`git config --global user.email "email`
Это инфа нужна в коммитах. 
Параметр `--global` - значит что эти настройки будут применены для всех репозиториев пользователя в системе
### Концепция рабочего процесса
#### Три состояния
Git оперирует тремя основными областями, через которые проходят изменения:
- **Рабочая директория** — здесь происходит непосредственная работа с файлами
- **Область подготовки (индекс)** — промежуточная область для формирования коммита
- **Репозиторий** — постоянное хранилище зафиксированных изменений
Изменения перемещаются между этими областями с помощью команд Git.
#### Жизненный цикл файлов
Каждый файл в рабочей директории может находиться в одном из состояний:
- **Неотслеживаемый (Untracked)** — Git не знает об этом файле
- **Отслеживаемый неизменённый (Unmodified)** — файл под контролем версий, не изменялся с последнего коммита
- **Изменённый (Modified)** — файл был изменён после последнего коммита
- **Подготовленный (Staged)** — изменения подготовлены для включения в следующий коммит

```1c
Жизненный цикл файла:

Неотслеживаемый → Добавление → Подготовленный
                                      ↓
Неизменённый ← Фиксация (коммит) ←────┘
      ↓
Изменённый → Подготовка → Подготовленный
```
#### Типичный рабочий процесс
Стандартный цикл работы выглядит следующим образом:
1. **Внесение изменений** — редактирование файлов в рабочей директории
2. **Просмотр состояния** — проверка, какие файлы изменены
3. **Подготовка изменений** — выбор файлов для включения в коммит
4. **Фиксация изменений** — создание коммита с описанием
5. **Повторение цикла** — возврат к пункту 1
### Структура Git репы
- **objects/** - база данных всех объектов(файлы, коммиты, деревья)
- **refs/** - ссылка на объекты коммитов(ветки, теги)
- **HEAD** - указатель на текущую ветку
- **index** - инфа об области подготовки
- **config** - конфиг репы
- **hooks/** - скрипты для автоматизаций
### Первые шаги после создания репозитория
После создания репозитория типичные первые действия включают:
- Создание файла README с описанием проекта
- Добавление файла .gitignore для исключения ненужных файлов
- Создание первого коммита с начальной структурой проекта
- Настройка удалённого репозитория (при необходимости)
