#### 1.2 Применить шаблон nginx.conf.j2 (путь: ../templates/nginx.conf.j2, не забудьте backup и notify)

``` YAML 
    - name: Настройка Nginx из шаблона 
      template: 
        src: "../templates/nginx.conf.j2"
        dest: /etc/nginx/sites-available/{{ app_name }}
        backup: yes
        validate: 'nginx -t -c %s'
      notify: Reload Nginx
      vars: 
        app_name: myapp
        nginx_port: 80
        ssl_enabled: true 
        
    - name: Включить конфиг nginx
      file:
        src: /etc/nginx/sites-available/myapp
        dest: /etc/nginx/sites-enabled/myapp
        state: link
      become: true
      
 handlers:
    - name: Проверить nginx конфиг
      command: nginx -t
      changed_when: false
      
    - name: Reload Nginx
      service:
        name: nginx
        state: reloaded
```
Объяснение: 
    - name: Настройка Nginx из шаблона  
      `template:`
		## Модуль Ansible который работает с шаблонами Jinja2
			Модуль это отдельная логика ansible. которая выполняет конкретную задачу #module_ansible 
			Jinja2 - это шаблонизатор 
        `src: template/nginx.conf.j2`
	        ##Путь до шаблона 
        `dest: /etc/nginx/sites-available/{{ app_name }}`
	        ## dest - Путь на сервере куда Ansible создать итоговый файл
		        {{ app_name }} - переменная из шаблона j2, подставляется из vars(что ниже)
        `backup: yes`  
	        ##Создать бэкап старого файла перед заменой 
		        Формат бэкапа /etc/nginx/sites-available/myapp.20251130-21:00~ 
        `validate: 'nginx -t -c %s'`
	        ##Проверяет синтаксис нового конфигурационного файла перед записью
		        nginx - команда запуска веб-сервера nginx
		        -t - test(проверка конфига)
		        -c - указать путь к файлу конфига 
		        %s - специальный placeholder, который Ansible заменяет на путь к dest 
			        placeholder - временная метка или что то по типу того, в данном контексте хранит в себе путь к файлу 
		    Нужно: если конфиг кривой ansible не перезапишет файл и Nginx не сломается
      `notify: Reload Nginx`
	      ## notify - выполняется только тогда когда таска ЧТО ТО МЕНЯЕТ(в данном случае конфиг nginx ), нужно это для того чтобы если таска ничего не поменяла в конфиге не перезапускать зазря Nginx 
	      P.S Notify берёт команду из `handlers` что находися в самом низу плейбука(НЕ ЗАБУДЬ ДОБАВИТЬ ТАСКУ С RELOAD NGINX В HANDLARS)
	            handlers:
		          - name: Reload Nginx
				 service:
				        name: nginx
			            state: reloaded       
      `vars:` 
        app_name: myapp
        nginx_port: 80
        ssl_enabled: true 
	        ##SSL - протокол для защищенного обмена данными между браузером и сервером
	    ## Внутри шаблона есть переменные и вот мы vars-ом даём этим переменным значения
	`name: Включить конфиг nginx` 
		Все строки в этом таске нужны чтобы тот создать симлинк нашего конфиг файла в папку -
		**sites-enabled**
		Если мы так не сделаем нихуя не будет работать т.к напомню *nginx* конфиги сайтов берет из -
		**sites-enabled**
	`state: link`
		**state** - его статус
		**link** - ссылка
	`changed_when: false` - говорит что ничего не менялось в системе не нужно записей или ещё чего то 
 
#### 1.3 Добавить health check (модуль uri с retries)
``` yaml 
    - name: Попытка подключения к внешнему API
      uri:
       url: "http://localhost:{{ web_port | default(80) }}"
       method: GET
       return_content: no
      retries: 3
      delay: 10
      register: api_check
      until:
        - api_check.status is defined
        - api_check.status == 200
      failed_when: false
      check_mode: no
      
    - name: Действие при недоступности API
      debug:
         msg: "API недоступен, используем локальную конфигурацию"
      when: api_check.status != 200
```
Объяснение: 
	- name: Попытка подключения к внешнему API
	  uri: 
		## uri - Модуль который делает HTTP(S) запрос
		    Дословно сходи по URL
		url: "https://api.example.com/health"
	    status_code: 200 
		    ## ожидаемый код если 200 збс
	  retries: 3 ## попытки
	  delay: 10 ## секунды ожидания между попытками
	  register: api_check 
		  ## Сохранить значения в переменную api_check 
		  Эта переменная используется ниже в строке when:
	  failed_when: false
	  `check_mode: no` # Нужно при запуске плейбику в чек моде т.к uri не поддерживает запуск в чек моде и выдаст ошибку при проверке
	.
	- name: Действие при недоступности API

	    msg: "API недоступен, используем локальную конфигурацию"
	  when: api_check.status != 200

##### Установка на CentOS
``` yaml 
- name: Добавление репозитория Docker на CentOS
  yum_repository:
    name: docker-ce
    baseurl: https://download.docker.com/linux/centos/7/x86_64/stable/
    gpgcheck: yes
    gpgkey: https://download.docker.com/linux/centos/gpg

```
Объяснение:

		rpm пакеты - формат пакетов для RedHat
	``gpgcheck: yes`` - проверка GPG подписей пакетов
	``gpgkey:`` URL ключа для проверки подписи пакетов
		Вообще gpg это ключ к подписи пакета
		пакет имеет подпись 
		вот и сверяется подпись с ключём правильно или нет #gpg
	``enabled: yes`` - репа активна и используется при установке пакетов 
#### 3.2 Node.js через NodeSource или nvm
##### NodeJs на Ubuntu
``` yaml
- name: Установка Node.js через NodeSource
  hosts: all
  become: yes

  tasks:
    - name: Установка curl (если нет)
      apt:
        name: curl
        state: present
        update_cache: yes

    - name: Добавление репозитория NodeSource для Node.js 20
      shell: curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
      args:
        executable: /bin/bash

    - name: Установка Node.js
      apt:
        name: nodejs
        state: present
        update_cache: yes
```
Объяснение: 
	``name: curl`` - Здесь один пакет — `curl` — инструмент для скачивания файлов через HTTP/HTTPS.
	 ``shell: curl -fsSL https://deb.nodesource.com/setup_20.x | bash -``
		 **shell:** модуль для выполнения команд через shell. Используется когда нельзя обойстись стандартными модулями Ansible
		 **`curl`** — скачивает файл по URL.
		 **`-f`** → fail silently при ошибке HTTP (код ошибки 4xx/5xx).
		 **`-s`** → silent, без прогресса загрузки.
		 **`-S`** → показывать ошибки curl, если есть.
		 **`-L`** → следовать за редиректами.
			 **Редирект** - значит перенаправление на другую ссылку
		 **`| bash -`** → передаёт скачанный скрипт сразу в bash для выполнения.
	``executable: /bin/bash`` - важная команда говорит используй именно **интерпретатор** - bash
	Теперь объяснялка ваще нахуя так сложно 
		**shell** через **curl** добавил оф репозиторий Node Js
		Сделано это было т.к в убунту устаревшие версий node js 
		А **apt** ВСЕГДА выбирает тот репозиторий в котором более новые версий(конечно из доступных ему репозиториев а доступен он т.к мы его добавили ПЛЮС nodesource автоматом добавил **GPG** ключ(ключ проверки пакета))
		**NodeJs** #nodejs - нужен для запуска приложений JavaScript вне браузера 
		**NodeSource** #nodesource - сторонний официальный дестребьютор nodjs. Он предостовляет:
			apt - репозиторий для Debian
			GPG - ключи для проверка пакетов
			Скрипт установки setup_XX.x который:
				определяет дистрибутив
				добавляет правильные репозиторий
				добавляет ключи
				готовит систему

#### 3.3 Установить yarn глобально (модуль npm)
``` yaml
- name: Установка yarn 
  npm: 
	  name: yarn 
	  state: present 
	  global: yes
```
Объяснение:
	**npm** - менеджер пакетов для Node.js
	Нужен он для: 
		Скачивание пакетов
		Установка их в проект 
		Управлять версиями пакетов
		запускать скрипты
	Состоит он из двух частей:
		**npm registry** - онлайн магазин пакетов для Node.js
		**npm CLI** - инструмент который работает с этими пакетами
	**yarn** - более лучшая альтернатива **npm CLI** 
	``global: yes`` - означает то что можно его использовать по всем системе а не только в каком нибудь проекте 
		Идеальная аналогия
		- **Локально** = "купил игрушку только для своей комнаты"
		- **Глобально** = "купил игрушку для всей квартиры, любой может взять её и играть"
