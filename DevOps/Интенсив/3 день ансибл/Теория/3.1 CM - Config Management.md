## 📌 Конфигурация

**Конфигурация** — это набор настроек, параметров и файлов, которые определяют, **как работает система, сервис или приложение**.

### В конфигурацию входит:

- **Системные настройки** (сеть, параметры ОС, системные файлы)
- **Установка и версии программ**
- **Пользователи и права**: группы, пароли, SSH-ключи
- **Конфигурационные файлы сервисов** (например, `nginx.conf`)
- **Переменные окружения** и **параметры командной строки**
- **Запуск и состояние служб** (автозапуск сервисов, например Docker)
- **Правила доступа**: firewall, политики SELinux
    - _SELinux решает, что можно, а что нельзя сервисам и пользователям внутри Linux_

---

## 📌 Конфиг-менеджмент (Configuration Management)

**Конфигурационный менеджмент** — это контроль и автоматизация применения конфигураций, чтобы инфраструктура работала **стабильно, безопасно и одинаково**.

Включает концепцию **IaC (Infrastructure as Code)**:
- Конфигурации описываются в виде кода
- Код автоматически применяется к серверам
- Серверы **не нужно настраивать вручную**

### Зачем нужен конфиг-менеджмент:
- Отслеживание всех изменений
- Автоматическая поддержка нужных конфигураций
- Стабильная работа систем
- Консистентность настроек на всех окружениях

---

## ⚠️ Без конфиг-менеджмента:

- **Дрейф конфигураций** — системы постепенно отходят от эталона
- **Человеческий фактор** — ошибки, забытые конфиги, разные настройки
- **Проблемы масштабирования** — нереально вручную настроить сотни серверов

---

## ✅ С конфиг-менеджментом:

- **Автоматизация** всех повторяющихся задач
- **Версионирование и прозрачность** — история изменений в Git
- **Воспроизводимость** — одинаковый результат при каждом выполнении
- **Скорость** — быстрое развёртывание и обновления систем

---

## 🎯 Примеры задач Configuration Management

- Массовое обновление **500+ серверов**
- Развёртывание новой среды из готового конфига (полностью идентично)
- Автоматические отчёты о состоянии систем
- Полная история изменений и авторов (через Git)

---

## 🧰 Инструменты конфиг-менеджмента

### **Ansible**
- Agentless — _не требует установки агентов на серверах_
- Работает через **SSH**
- Использует **push-модель**: центральный сервер _толкает_ конфиг на целевые машины
- Основная сущность — **playbook**

### **Terraform**
- Инструмент для создания инфраструктуры
- Особенно полезен в облаках и больших мультиоблачных проектах
- Управляет виртуалками, сетями, БД, балансировщиками, кластерными ресурсами

# 🧠 Ключевые принципы IaC

### **Декларативность**

> Ты описываешь _какой результат нужен_, а не _какие шаги выполнять_.

Пример: "Пакет nginx должен быть установлен".

### **Идемпотентность**

> Многократное выполнение даёт **одинаковый результат**.

Пример: запуск одного и того же playbook 10 раз — конфигурация остаётся одинаковой.

| Характеристика          | Configuration Management (CM)                                                                    | Infrastructure as Code (IaC)                                                               |
| ----------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| Основная цель           | Управляет конфигурациями **существующих систем**                                                 | **Создает саму инфраструктуру** из кода.                                                   |
| **Что делает**          | Настраивает системы — устанавливает ПО, применяет конфигурации, обеспечивает желаемое состояние. | Создает IT-ресурсы: виртуальные машины, сети, балансировщики нагрузки.                     |
| **Эволюция**            | Является основой для автоматизации.                                                              | Является эволюцией CM.                                                                     |
| **Инструмент (Пример)** | Ansible (использует Плейбуки).                                                                   | Terraform (использует HCL).                                                                |
| **Принципы**            | Обеспечивает консистентные (одинаковые) настройки во всех окружениях.                            | Использует декларативность и идемпотентность для предсказуемости и безопасности изменений. |
|                         |                                                                                                  |                                                                                            |
