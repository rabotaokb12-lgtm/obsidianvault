#Flood 
	«широковещательная рассылка для неизвестного MAC» если пакет был отправлен на неизвестный mac тогда тогда свич отправляет пакет всем и если находит mac получателя(PC2) записывает его в свою таблицу и mac получателя получает свой пакет. Если же пакет попал не в тот mac то РСЗ говори MAC не мой, дропаю
	PC2 (нужный адресат) принимает кадр, так как **MAC назначения совпадает с его MAC**.
	PC3 (не адресат) тоже **получает копию**, но его сетевой адаптер смотрит заголовок Ethernet:
	-если **MAC назначения = его MAC** → кадр обрабатывается;
	-если **MAC не совпадает** → адаптер даже не передаёт кадр в ОС, он просто**отбрасывается на уровне сетевой карты**.

**NIC (Network Interface Card)** — это **сетевая карта**.

#marshrutizacia 
	**Маршрутизация** — это процесс **доставки пакета из одной сети в другую**
	Без неё L2-Отправляет пакеты через mac ЕСЛИ они находятся в одной подсети(IP нет)
	С маршрутизацией отправляются  пакеты из разных подсетей(нужен ещё IP)
	КАК именно это работает:
         ПК1(192.168.10.10) хочет передать пакет на ПК2(192.168.20.20)
        1.**Пакет** создаётся упаковывается в **кадр** на ПК1 и отправляется через шлюз VLAN10 к роутера
        2.Роутер получает кадр смотрит ip и создав новый кадр отправляет его на на ПК2 через шлюз VLAN20
        Примечание:
        ВСЕ кадры идут через L2 уровень а пакеты на L3 
        У каждого l2 свича есть таблица MAC адресов
        l2 свичи могут видеть vlan-ы т.к к каждый vlan присвоен к какому то порту + у vlan может быть mac адрес если он взаимодействует с l3 уровнем(в нашем случае это роутер)
        **Кадры** отправляются на mac и по mac определяется через какой **ПОРТ** VLAN он пойдёт

#arp
     **ARP (Address Resolution Protocol)** — это протокол, который «переводит» IP-адрес в MAC-адрес.
     Кароче это работает так чтобы отправить пакет в разных подсетях
         ПК1(192.168.10.10) хочет передать пакет на ПК2(192.168.20.20)
         1.ПК1 кричит в сеть(**broadcast**) у кого IP 192.168.20.20? ДАЙ MAC
         2.ПК2 отвечает своим MAC
         3.Компьютер сохраняет это в таблицу ARP и дальше знает, к какому MAC посылать кадры
    Для понимания внутри LAN ПК1 хочет отправить данные на ПК2
        1.Создаётся **IP-пакет** туда засовывают ip двух компом
        2.Этот пакет упаковывается в **кадр Ethernet**: mac ПК1, mac шлюза VLAN10
        3.Роутер получает кадр, проверяет IP. создаёт новый **кадр**:mac ПК2, mac шлюза VLAN20
        4.Новый кадр отправляется на ПК2 по VLAN20
        5.ПК2 получает кадр и отправляет ответ в таком же порядке на ПК1
        ВСЕ кадры идут через L2 уровень а пакеты на L3

#nat 
     **NAT (Network Address Translation)** — это технология, которая позволяет менять IP-адреса пакетов при прохождении через маршрутизатор или брандмауэр.
     Проще говоря, NAT нужен, чтобы:
    -**Много устройств** в локальной сети могли выходить в интернет через **один внешний IP**.
    -Скрывать внутренние адреса (безопасность).
    -Экономить "белые" IPv4-адреса, которых мало.
    **Как работает**:
    1.Ноутбук с ip 192.168.0.10:51512 отправляет пакет на сервер google.com
    2.Роутер с NAT принимает пакет и видит что пакет с частного адреса
    3.NAT записывает в таблицу что его внешний ip:порт  92.184.11.55:40001 равен внутреннему 192.168.0.10:51512 
    3.Отправляет пакет на google.com с внешнего ip и портом
    4.Получает ответный пакет и сверяясь с таблицей и отправляет этот пакет обратно на внутренний ip
    Внешние порты в данном контексте нужны чтобы сверять какому внутреннему ip отправить встречный пакет

#firewall 
    **Фаервол** — это система фильтрации сетевого трафика.  
    Она решает: **какие пакеты разрешить, а какие заблокировать**, исходя из набора правил.
    **Как** работает фаервол:
    1. **Пакет приходит** на сетевой интерфейс устройства (роутер, сервер, ПК).
    2. **Фаервол сравнивает** его параметры с правилами:
    - IP источника и назначения
    - Порт источника и назначения (например, 80 = HTTP, 443 = HTTPS, 22 = SSH)
    - Протокол (TCP, UDP, ICMP)
    - Направление (входящий, исходящий, пересылаемый)
    1. **Действие по правилу**:
    - `ACCEPT` → пропустить 
    - `DROP` → молча выкинуть
    - `REJECT` → отказать и уведомить отправителя

#dhcp_discover #broadcast #unicast #relay
	Так сперва объясню это всё связанный процесс выдачи dhcp
	1.Комп даёт запрос на ip
	2.DHCP-запрос(discover) всегда идёт в broadcast 255.255.255.255
	3.**Broadcast** - кричит выдайте ip, по всей сетке и по всем портам  
	    Как клиент без ip может достучаться до broadcast, легко если он подключен к сетке но у него нет ip то ему выдаётся 0 ip и этого хватает чтобы дальше идти по схеме
	4.Возможно broadcast кричит в свич или роутер на котором l3 и выше уровня 
    4.1.Тогда **relay** берёт этот крик и **unicast**-ом отправляет этот крик на dhcp сервер(relay знает по какому ip находиться dhcp server)
         **relay** - помощник который превращает крик в письмо и обратно(на всём пути                     хранит в себе mac клиента)
         **unicast** - то самое письмо, он от relay улетает с:
            -IP-источник: адрес VLAN интерфейса (например 192.168.1.1). это карочь vlan      ip свича с которого **relay** отправляет запрос
            -IP-назначение:адрес DHCP-сервера(например 10.10.10.5).
            -Внутри полезной нагрузки он сохраняет оригинальный DHCP Discover клиента.
	4.2.ЕСЛИ broadcast кричит на l2 свич тогда l2 **flood**-ит на все остальные vlan и там точно найдётся свич более высоко уровня и дальше идёт пункт 4.1
	5.DHCP сервер получает письмо от **relay** и отправляет **dhcp offer** обратно в котором:
	    -**IP-источник:** IP DHCP-сервера (например, `10.10.10.5`)
        -**IP-назначение:** IP relay (например, `192.168.1.1`)
        -**DHCP-поля:**
            - `yiaddr` (your IP address) = IP, который сервер предлагает клиенту (например, `192.168.1.100`)
            - `giaddr` (gateway IP address) = адрес relay (тот же `192.168.1.1`), чтобы сервер знал, куда слать
    6.**Relay** снова **broadcast**-кричит на все vlan-ы чтобы клиент узнав свой mac принял этот **dhcp offer** (так же работает если клиент подключен к l2 свичу)
